# UI Pattern Library Design

**Status:** Approved
**Date:** 2026-04-11
**Related roadmap item:** Phase 3 — Richer UI component templates

## Goal

Raise the baseline quality, visual consistency, and compile reliability of utility apps generated by VibeApp, without constraining the AI's imagination in creative scenarios.

## Context

Generated apps today suffer from three recurring issues:

1. **Low-quality layouts.** The LLM writes XML from scratch each time, inventing spacing, typography, and component combinations. Results look amateurish even when they compile.
2. **Inconsistent aesthetics.** No shared design system — two apps built back-to-back feel like different products.
3. **Compile fragility.** Hand-written XML often breaks AAPT2 (wrong attrs, missing themes, bad component nesting).

The `agent-system-prompt.md` currently contains only a 10-line "UI Tips" section (emoji, simple vector drawables, system bars). There is no layout pattern library, no component recipe catalog, no design token reference.

## Non-Goals

- **Brand imitation.** Unlike VoltAgent's `awesome-design-md`, we do not offer "Linear style" or "Notion style". One consistent Material 3 utility aesthetic.
- **Creative override of structure.** Creative Mode relaxes palette and typeface but still enforces spacing/corner/elevation whitelist and ShadowActivity/bundled-lib rules.
- **Compose templates.** Generated apps remain Java + XML. No Compose in the pattern library.
- **Third-party design kits.** No paid UI kits, no Figma exports.

## Inspirations

- [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) — DESIGN.md-as-markdown concept. Borrowed: markdown delivery, LLM-friendly prose format. Not borrowed: brand-flavored variation, CSS/Tailwind tokens.
- [material-components-android/catalog](https://github.com/material-components/material-components-android) — Apache 2.0 component usage reference. Borrowed: structural correctness of individual component usage. Not borrowed: full files (bundled lib is a subset).
- [Material 3 spec (m3.material.io)](https://m3.material.io) — open design tokens. Borrowed: baseline palette, type scale, spacing, shape, elevation.

## Architecture

```
┌─────────────────────────────────────────────────┐
│  System Prompt (embedded, ~300 words)           │
│  - Hard token whitelist (colors/spacing/type/   │
│    corner/elevation)                             │
│  - Tool decision flow                            │
│  - Creative Mode triggers                        │
└─────────────────────────────────────────────────┘
           │ fetches on demand
           ▼
┌────────────────────┐    ┌────────────────────┐
│ UI Pattern Library │    │   Design Guide     │
│ (assets/patterns/) │    │ (assets/design/)   │
│ - 14 component     │    │ - Full markdown    │
│   blocks           │    │ - Sectioned by H2  │
│ - 5 screen         │    │                    │
│   templates        │    │                    │
│ - index.json       │    │                    │
└────────────────────┘    └────────────────────┘
           │                      │
           └──────┬───────────────┘
                  ▼
        ┌──────────────────┐
        │  Agent Tools     │
        │ - search_ui_pattern
        │ - get_ui_pattern │
        │ - get_design_guide
        └──────────────────┘
```

**Three-layer division of labor:**

1. **Tokens (embedded)** enforce aesthetic consistency. A baseline palette, type scale, spacing whitelist, corner radius whitelist, elevation whitelist. Always in context.
2. **Pattern library (on-demand)** provides compilable building blocks and a small set of screen templates. Fetched by tool calls.
3. **Design guide (on-demand, full version)** provides usage rules, creative mode semantics, and detailed examples. Fetched by tool call when the AI needs deeper reference.

**Core principles:**

- The pattern library is a **floor, not a ceiling**. It prevents broken XML and ugly token-less spacing; it does not dictate what "good" looks like.
- **Tokens carry consistency; templates carry efficiency.** Two apps can have totally different structures yet feel cohesive because they share tokens.
- **Fetched patterns must be adapted, never pasted verbatim.** Change copy, remove unused slots, rearrange.
- **Creative Mode** lets the AI skip the library and override palette/typeface when the user asks for expressive output, while still respecting structural constraints.

## Data Structures

### meta.json (human-authored, one per pattern)

```json
{
  "id": "list_item_two_line",
  "kind": "block",
  "description": "Standard two-line list item: icon + title + subtitle + trailing control",
  "keywords": ["list", "item", "row", "列表", "条目"],
  "slots": [
    { "name": "title", "description": "Primary title text", "default": "Item title" },
    { "name": "subtitle", "description": "Secondary text", "default": "Item subtitle" },
    { "name": "icon", "description": "Leading icon drawable reference", "default": "@android:drawable/ic_menu_info_details" }
  ],
  "dependencies": []
}
```

### layout.xml (with `{{slot}}` placeholders)

```xml
<LinearLayout xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"
    android:orientation="horizontal"
    android:minHeight="64dp"
    android:gravity="center_vertical"
    android:paddingHorizontal="16dp"
    android:paddingVertical="8dp">

    <ImageView
        android:layout_width="40dp"
        android:layout_height="40dp"
        android:src="{{icon}}"
        android:contentDescription="@null"/>

    <LinearLayout
        android:layout_width="0dp"
        android:layout_height="wrap_content"
        android:layout_weight="1"
        android:orientation="vertical"
        android:paddingStart="16dp">

        <TextView
            android:text="{{title}}"
            android:textAppearance="?attr/textAppearanceTitleMedium"
            android:layout_width="wrap_content"
            android:layout_height="wrap_content"/>

        <TextView
            android:text="{{subtitle}}"
            android:textAppearance="?attr/textAppearanceBodyMedium"
            android:textColor="?attr/colorOnSurfaceVariant"
            android:layout_width="wrap_content"
            android:layout_height="wrap_content"/>
    </LinearLayout>
</LinearLayout>
```

### index.json (build-time generated)

```json
{
  "version": 1,
  "generated": "2026-04-11T10:00:00Z",
  "patterns": [
    {
      "id": "list_item_two_line",
      "kind": "block",
      "description": "...",
      "keywords": ["list", "item", "row"],
      "slotNames": ["title", "subtitle", "icon"],
      "dependencies": []
    }
  ]
}
```

Index contains only metadata. Layout XML and notes are loaded on demand by `get_ui_pattern`.

### Kotlin types

```kotlin
enum class PatternKind { BLOCK, SCREEN }

data class PatternSlot(
    val name: String,
    val description: String,
    val default: String,
)

data class PatternRecord(
    val id: String,
    val kind: PatternKind,
    val description: String,
    val keywords: List<String>,
    val slots: List<PatternSlot>,
    val dependencies: List<String>,
    val layoutXml: String,
    val notes: String,
)

data class PatternSearchHit(
    val id: String,
    val kind: PatternKind,
    val description: String,
    val keywords: List<String>,
    val slotNames: List<String>,
    val dependencies: List<String>,
)
```

### Search ranking (PatternSearch)

1. Exact `id` match → highest
2. `id` substring match
3. `keywords` hit
4. `description` substring match
5. Tie-break by `id` alphabetical order

Case-insensitive. Supports Chinese and English keywords.

## Agent Tools

### Tool 1: `search_ui_pattern`

**Input:**
- `keyword: string` (required)
- `kind: "block" | "screen" | "any"` (default `"any"`)
- `limit: int` (default 10, range 1..30)

**Output:** `{ total: int, hits: [{ id, kind, description, keywords, slotNames, dependencies }] }`

**Errors:** empty keyword → `"keyword required"`

**Behavior:** Load `index.json`, rank per rules, return hits without layoutXml/notes (save tokens).

### Tool 2: `get_ui_pattern`

**Input:** `id: string` (required)

**Output:** `{ id, kind, description, layoutXml, slots[], usageNotes, dependencies[] }`

**Errors:** unknown id → `"Unknown pattern id '<id>'. Call search_ui_pattern first."`

**Behavior:** Return full `PatternRecord`. Caller decides how to write into the project.

### Tool 3: `get_design_guide`

**Input:** `section: "tokens" | "components" | "layout" | "creative" | "all"` (default `"all"`)

**Output:** `{ section, content }`

**Errors:** invalid section → error listing valid options

**Behavior:** Read `assets/design/design-guide.md`, slice by H2 heading.

## System Prompt Changes

**Remove:** existing "UI Tips" section.

**Keep:** "System Bars & Window Insets" (compile-safety concern).

**Add** "Design Guide (Embedded Hard Constraints)" section:

```markdown
## Design Guide (Embedded Hard Constraints)

Tokens (whitelist — violations are errors):
- Colors: use M3 baseline attrs only — ?attr/colorPrimary, ?attr/colorOnPrimary,
  ?attr/colorSecondary, ?attr/colorTertiary, ?attr/colorSurface,
  ?attr/colorSurfaceVariant, ?attr/colorOnSurface, ?attr/colorOnSurfaceVariant,
  ?attr/colorError. No hex literals unless Creative Mode.
- Spacing: pick from 4 / 8 / 12 / 16 / 24 / 32 (dp).
- Text: ?attr/textAppearanceDisplayLarge / HeadlineLarge / TitleLarge /
  TitleMedium / BodyLarge / BodyMedium / LabelMedium.
- Corner radius: 4 / 8 / 12 / 16 / 28 (dp).
- Elevation: 0 / 1 / 3 / 6 (dp).
- Screen horizontal padding defaults to 16dp.
- Touch target ≥48dp.

Hard rules:
- Always use MaterialToolbar as a regular View, never setSupportActionBar().
- RecyclerView item spacing via padding, not ItemDecoration.
- Form row height ≥48dp.
```

**Add** "UI Pattern Library" section:

```markdown
## UI Pattern Library

Tools: search_ui_pattern / get_ui_pattern / get_design_guide

Decision flow when building UI:
1. Is the request creative (triggers: 好看 / 有设计感 / 复古 / 童趣 / 酷炫 /
   极简 / 暗黑 / 像 ___ 一样)?
   YES → skip library, use embedded tokens. Allow overriding
         primary/secondary palette and typeface.
   NO  → continue.
2. Is it a standard utility screen (list/form/settings/detail/dashboard)?
   YES → search_ui_pattern(keyword, kind="screen") as a shortcut.
   NO  → search_ui_pattern(keyword, kind="block") and compose.
3. Uncertain about tokens or component rules?
   → get_design_guide(section="...").
4. ALWAYS adapt fetched patterns — change copy, remove unused slots,
   rearrange. NEVER paste verbatim. The library is a floor, not a ceiling.

Slot format: layoutXml contains {{slot_name}} placeholders. Replace every one
with a real value from slots[].default or something task-specific.
```

## Compile Smoke Testing

**Test location:** `build-engine/src/test/java/com/vibe/build/engine/resource/PatternCompileSmokeTest.java`

**Flow:**

1. Locate `app/src/main/assets/patterns/` (relative to build-engine test working dir).
2. Enumerate all `blocks/*/layout.xml` and `screens/*/layout.xml`.
3. For each layout:
   1. Parse meta.json → extract `slots[]`.
   2. Replace every `{{slot_name}}` in the XML with the slot's `default` value → instantiated XML.
   3. Build a minimal Android resource project in a temp dir:
      ```
      tempDir/
        res/
          layout/pattern_under_test.xml   ← instantiated XML
          values/themes.xml               ← copied from template
          values/colors.xml               ← copied from template
          values/strings.xml              ← empty
      ```
   4. Invoke build-engine's existing AAPT2 wrapper (`Aapt2Compiler` or equivalent) on the temp dir.
   5. Assert exit code 0, empty stderr.
4. Any failure → report pattern id + instantiated XML + AAPT2 stderr.

**Test context sources:**

- `themes.xml` and `colors.xml` copied from `app/src/main/assets/templates/EmptyActivity/app/src/main/res/values/` so patterns resolve `?attr/textAppearanceTitleMedium`, `?attr/colorPrimary`, etc.
- `android.jar` obtained from the build-engine test classpath.

**AAPT2 binary resolution:**

- Reuse whatever build-engine already uses at test time. If build-engine doesn't currently invoke AAPT2 in tests, Phase C's first task is a spike to find out.
- **Fallback:** if CI lacks AAPT2, the test skips with a loud warning rather than failing. Local dev environment must have it.

**Fallback if spike fails:** degrade to `PatternXmlValidityTest` (XmlPullParser parse + shallow schema checks) plus a `docs/ui-pattern-manual-smoke.md` describing manual verification.

**`PatternIndexGeneratorTest` location:** `app/src/test/kotlin/...` (no AAPT2 needed). Side effect: rewrites `app/src/main/assets/patterns/index.json`. Test comment must document "running this test regenerates index.json".

## Known Risks

1. **build-engine can't invoke AAPT2 standalone** → Phase C has fallback to XML validity checks + manual smoke doc.
2. **AI forgets to replace `{{slot}}`** → system prompt hard rule + usage notes + future: lint generated XML for unreplaced `{{}}`.
3. **Material 3 attrs missing from bundled themes** → Phase D first task validates that `Theme.MyApplication` resolves every `?attr/...` used. If not, either downgrade to concrete values or patch `themes.xml`.
4. **14 blocks prove too few** → structure supports adding new directories + re-running `PatternIndexGeneratorTest`. No code changes.
