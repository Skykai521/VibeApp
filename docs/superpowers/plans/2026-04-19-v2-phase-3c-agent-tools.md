# v2.0 Phase 3c: Agent tool chain for v2 projects

**Goal:** give the chat agent a complete set of tools to create, edit,
build, install, modify-dependencies-of, and export Kotlin + Compose
projects via the on-device Gradle pipeline established in Phases 2a–2e
+ 3a/3b. After Phase 3c, "make a counter app" → agent runs
`create_compose_project` → `assemble_debug_v2` → `install_apk_v2`
end-to-end.

**Architecture:** Six new `AgentTool` implementations under
`app/src/main/kotlin/com/vibe/app/feature/agent/tool/`, each scoped to
the v2 GRADLE_COMPOSE engine and refusing v1 (LEGACY) projects with an
explicit error message. The Project Room entity gains an `engine`
column so v1 and v2 projects can coexist; the Project's `workspacePath`
is engine-aware (v1 = `.../{id}/app`, v2 = `.../{id}` repo root). The
existing `read/write/list/grep_project_files` tools work unchanged for
both engines because they all read `workspacePath` directly. Net new
DB schema is one column + one migration; net new Kotlin is one
ProjectManager method + six tool classes + DI bindings.

## Scope

## Exit Log (2026-04-19)

**Validated by:**
- `:app:kspDebugKotlin` (full Hilt + Room codegen): green.
- `:app:assembleDebug`: green.
- DB migration: not directly executed (no on-device upgrade run), but the SQL is a single non-destructive `ALTER TABLE ... ADD COLUMN ... DEFAULT 'LEGACY'`. Room's exported schema 4.json validates.
- Agent integration: tools are registered + visible to the registry. End-to-end "agent says 'make a counter' → real APK on screen" exercise belongs to Phase 3d (chat UX wiring).

**Notable design decisions:**

1. **`engine` lives on the Project row, not on a separate "v2 project" table.** Keeps the existing `Project` queries / DAO API working unchanged; v1 vs v2 logic is opt-in per consumer.
2. **`workspacePath` is engine-aware.** v1 projects keep their existing `.../{id}/app` semantics (`rootDir` = the app/ subdir). v2 projects use `.../{id}` (repo root), which is what `GradleBuildService.runBuild` and the agent's file-tools both naturally want. No code-level branching needed in `DefaultProjectWorkspace`; the schema just stores a different value.
3. **6 separate `*V2Tool` classes rather than one monolithic "v2 helper" tool.** Each is < 150 lines, single-responsibility, separately testable, and surfaces in the agent's tool list with its own JSON schema (so the model can pick the right one).
4. **Atomic dep edits use in-memory rollback, not tmpfile + rename.** Both files have to land or neither — easier to reason about than two-phase-commit on disk. Both files are < 5 KB so the in-memory cost is negligible.
5. **`create_compose_project` chains chat → project, not chat-replaces-project.** A chat may already have a v1 project (from `init_project`); calling `create_compose_project` creates a NEW Project row tied to the same chatId. The Project schema already allowed multiple projects per chat (no unique constraint on `chat_id`), so this works without further schema changes. Phase 3d will add UI to pick which project a chat is currently "active" on.
6. **System prompt change is conservative.** v2 tools are described as "opt-in" / "use only when the user asks for Kotlin / Compose explicitly". The default Java + XML flow stays the recommended path until Phase 6 (legacy cleanup).

**Phase 3c is complete.** Phase 3d (chat UX: "New Compose Project" button, project switcher, run-progress UI) is unblocked and is what's needed to make all this user-visible.
